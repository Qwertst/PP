# Отчёт: Потокобезопасный IntSet на основе связного списка

## Стратегии синхронизации

### 1. Грубая (Coarse-grained)
Один мьютекс на весь список. Каждая операция (add, remove, contains) захватывает глобальный мьютекс. Простая реализация, но все потоки сериализуются — при высокой конкуренции становится узким местом.

### 2. Тонкая (Fine-grained)
Мьютекс на каждый узел. При обходе списка потоки захватывают мьютексы попарно: блокируют текущий и следующий узел, затем отпускают предыдущий. Позволяет нескольким потокам одновременно работать с разными частями списка.

### 3. Оптимистичная (Optimistic)
Поиск выполняется без блокировок. После нахождения нужной позиции захватываются мьютексы двух узлов и выполняется валидация — проверка, что узлы всё ещё в списке и связаны друг с другом. При неудачной валидации операция повторяется.

### 4. Ленивая (Lazy)
Аналогична оптимистичной, но с двумя ключевыми отличиями:
- Удаление двухфазное: сначала узел помечается флагом `marked`, затем физически отсоединяется.
- Валидация проще: достаточно проверить флаги `marked` и связь `prev->next == curr`.
- `contains()` полностью безблокировочный — просто обходит список и проверяет ключ и флаг.

## Условия бенчмарка

- Соотношение операций: 20% запись (10% add + 10% remove) / 80% чтение (contains)
- Диапазон ключей: 0–1023
- Длительность: 3 секунды на каждый тест
- Количество потоков: 1, 2, 4, 8

## Компиляция:

```bash
g++ -std=c++17 -O2 -pthread -o benchmark main.cpp
```

## Анализ результатов

### Грубая синхронизация
- Лучший результат на 1 потоке (7.5M ops/sec) — минимальные накладные расходы.
- Деградация с ростом потоков: 8 потоков дают лишь 1.3M ops/sec — все потоки ждут один мьютекс.

### Тонкая синхронизация
- Худший результат во всех конфигурациях (57K–457K ops/sec).
- Огромные накладные расходы на захват/освобождение мьютекса на каждом узле при обходе.

### Оптимистичная синхронизация
- Стабильная производительность (~1M ops/sec) при любом числе потоков.
- Масштабируется положительно: 1.9M ops/sec на 8 потоках.
- Двойной обход (поиск + валидация) ограничивает максимальную скорость.

### Ленивая синхронизация
- Близка к грубой на 1 потоке (7M ops/sec).
- **Единственная стратегия, которая масштабируется сверхлинейно**: 12M ops/sec на 8 потоках.
- Безблокировочный `contains()` — ключевое преимущество при 80% чтения.

## Выводы: когда «ленивая» синхронизация выигрывает

1. **Нагрузка с преобладанием чтения.** При 80% операций `contains()` безблокировочное чтение даёт колоссальное преимущество. Потоки не конкурируют за мьютексы при чтении.

2. **Большое количество потоков (4+).** Ленивая — единственная стратегия, показывающая рост производительности с увеличением числа потоков. На 8 потоках она в 9× быстрее грубой и в 6× быстрее оптимистичной.

3. **Средние и большие списки.** Чем длиннее список, тем дороже обход с блокировками (тонкая) или двойной обход (оптимистичная). Безблокировочный `contains()` ленивой стратегии выигрывает тем больше, чем длиннее список.

### Когда «ленивая» проигрывает

- На 1 потоке грубая синхронизация немного быстрее (~7%) из-за отсутствия накладных расходов на валидацию.
- При высокой доле записей (>50%) преимущество безблокировочного `contains()` нивелируется, и накладные расходы на двухфазное удаление становятся заметны.
